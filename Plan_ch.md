1.	实践目的
本课程旨在使学生能够将数据库系统的基础理论知识应用于实际问题的分析与系统设计中，独立完成一个完整的数据库应用系统。通过本次实践，学生应能够：
[1]	掌握数据库系统的基本原理与设计方法，理解并实践数据库的需求分析、概念结构设计、逻辑结构设计和物理结构设计全过程；
[2]	熟练运用主流数据库管理系统（如 MySQL、Oracle、SQL Server 等），掌握 SQL 语言的使用，能够完成数据库的创建、表结构设计、数据操作及复杂查询；
[3]	掌握数据库程序设计能力，能够编写并调试存储过程、存储函数、触发器等，实现数据一致性维护与业务逻辑控制；
[4]	具备数据库应用开发能力，能设计简洁易用的前端界面，与数据库实现有效交互，形成完整的系统功能；
[5]	培养独立思考与问题解决能力，提高资料查阅、文档撰写和系统实现的综合实践水平；
[6]	强化工程规范与质量意识，在项目实施过程中重视系统安全性、稳定性与可维护性。

本次实践需满足以下要求：
（一）前端设计要求：
（1）设计与交互性
界面应美观、布局合理，兼容 Chrome、Firefox、Edge 等主流浏览器，并能自适应不同分辨率。操作流程应简洁流畅，符合用户常规使用习惯。各功能模块应通过清晰的导航栏或菜单加以区分，便于用户快速定位并进入所需功能界面。

系统应具备必要的数据验证机制，在用户提交数据前，对输入内容的合法性、完整性和格式进行检查，防止不符合要求的数据写入数据库，从而保障系统的稳定性与数据的可靠性。

（二）数据库设计要求：
（1）需求与概念设计
能结合实际业务场景进行需求分析，明确系统中的实体、属性及其关系，绘制规范的 E-R 图，并将其转换为合理的关系模式。通过 1NF～3NF（或 BCNF）等规范化过程，确保数据结构的完整性与可维护性。
（2）逻辑与物理设计
能将概念模型转化为符合数据库管理系统要求的逻辑结构，合理确定主键、外键及约束条件；结合系统访问特征，设计索引、分区与存储方案，提升数据存储效率与查询性能。
（3）SQL 编程与性能优化
能熟练使用 SQL 进行数据库对象定义与操作，编写复杂查询语句（多表连接、聚合、子查询等），并针对性能瓶颈进行分析与优化，实现高效的数据管理。
（4）事务与安全管理
能设计并实现完整的事务逻辑，正确处理并发与异常情况，保障数据一致性与可靠性；同时具备数据库用户与权限管理能力，实施安全控制、加密与审计策略，确保系统合规安全。
（5）程序化与维护能力
能在数据库端编写存储过程、触发器、游标等程序对象以实现复杂业务逻辑；掌握数据备份、导入导出与恢复操作，具备数据库维护与持续运行保障的综合工程能力。

（三）其他要求：
（1）所有的源代码、数据库备份文件、实验报告、PPT、演示视频等都需提供，提前发送至指定邮箱； 
（2）代码符合代码规范要求。
4.	实践任务
该数据库系统需实现数据在不同数据库平台的统一存储与同步管理，同时具备完整的前端交互、权限控制与报表分析功能，包括：
（1）系统需同时支持不少于三种数据库系统（如 MySQL、Oracle、SQL Server 等），并实现跨库数据同步功能；
（2）所有数据库中的表结构保持一致，数据库表数量不少于 5 张（学生可根据系统需求自定义具体表内容）；
（3）系统需实现多数据库间的数据一致性维护，同步操作包括：
•	当任一数据库发生数据变更时（以考试组卷系统为例，需包括新增试题或更新成绩），系统应立将更新同步至其他数据库；
•	系统可根据配置周期（如每日/每小时）自动执行同步任务。
（4）用户与权限管理：设置不同操作权限；
（5）冲突处理：数据冲突时自动邮件提醒管理员，管理员可在 PC 端选择数据来源进行修复；移动端可查看每日同步统计与异常报表（图形化展示）。
（6）数据管理：支持数据分类、录入、修改、查询，支持多条件检索与复杂 SQL（多表连接、嵌套子查询）查询优化；
（7）信息管理：支持各类信息的记录。

## 在线图书馆管理系统策略计划书

### 1. **多数据库同步策略**：

* **同步机制**：

  * **实时同步**：每当数据库中的借阅记录、图书信息或用户数据发生变化时，立即将这些变化同步到其他数据库中。可以通过触发器（如 MySQL 中的 `AFTER INSERT/UPDATE`）实现实时同步。实时同步确保了所有数据库中的信息保持最新，避免了数据滞后的问题。

  * **定期同步**：使用定时任务（如 cron 作业）在预定时间（例如每天/每小时）执行数据同步，确保数据库的完整性和一致性。定期同步适用于较少变动的表数据，减少实时同步的负担。

  * **冲突检测与自动修复**：当系统发现两个数据库中的数据冲突时（如不同数据库中的借阅记录信息不一致），自动通过邮件通知管理员。管理员可以手动选择一个数据源进行修复，或让系统按规则自动解决（如优先保留最新数据）。为避免误操作，管理员将收到详细的冲突报告。

* **同步的性能优化**：

  * **增量同步**：每次同步仅传输变更数据，而非全量数据。这可以减少同步时间和带宽消耗。增量同步通过比较数据库中记录的时间戳或最后修改时间来仅同步需要更新的数据。

  * **数据去重**：在同步过程中进行数据去重，确保每条记录在每个数据库中都是唯一的，避免由于重复数据而导致的同步失败。

  * **同步日志**：记录每次同步的详细日志，确保系统能够追溯每次同步操作的结果和错误，提供审计功能。通过日志文件，管理员能够查看每个数据库的同步状态及历史操作，进一步加强数据管理的透明度和可靠性。

---

### 2. **用户权限管理**：

* **细粒度权限控制**：

  * **管理员**：拥有最高权限，能够访问系统中的所有数据和设置，执行所有操作，如新增/删除用户、管理图书馆设备、查看报表、执行系统同步等。

  * **图书管理员**：管理图书信息和借阅记录，可以修改图书信息、调整库存、查看借阅记录和发送逾期提醒。

  * **读者**：仅限于查询图书、借阅图书和查看自己的借阅历史。读者不能修改任何数据，保证数据安全性。

  * **动态权限控制**：根据需求增加临时权限，如在某些特定情况下允许读者修改自己的联系方式或借阅历史。

* **角色与权限分配**：

  * 设计权限系统（RBAC），基于角色控制用户的访问权限。管理员可以管理角色和用户的权限，确保每个用户只能执行允许的操作。

  * 使用数据库中的用户表（如 `users`、`roles`、`permissions`）来灵活配置和分配权限，确保每个用户只能访问授权的数据。权限管理中还包括对重要操作的审计功能，如数据删除、修改等敏感操作的记录和审查。

---

### 3. **图书管理**：

* **图书信息管理**：

  * **新增图书**：图书管理员可以通过表单或批量导入功能添加新书，包含书名、作者、出版社、ISBN、分类、库存数量等字段。

  * **图书信息修改与删除**：管理员或图书管理员可以修改现有书籍的详细信息，删除损坏或无法借出的图书。

  * **图书库存管理**：图书管理员可动态更新库存，处理损坏或丢失的书籍，确保库存信息的准确性。

* **智能推荐系统**：

  * 通过分析读者借阅历史和图书借阅热度，自动生成个性化的图书推荐列表。可以使用基于协同过滤（Collaborative Filtering）或内容推荐（Content-based Filtering）的方法来实现推荐系统。

* **图书分类与检索**：

  * 支持多种方式的图书检索，如按书名、作者、ISBN、出版日期、分类等进行检索。

  * 提供“快速检索”和“高级检索”选项，读者可以选择更多筛选条件，如语言、书籍类型等。

---

### 4. **借阅管理**：

* **借阅功能**：

  * **借阅申请**：读者可以选择书籍并提交借阅申请，图书管理员审批借阅请求。

  * **借阅限制**：系统可设定借阅规则，例如每个读者最多借阅多少本书、借阅期限等，超过限制时自动提醒。

  * **延迟还书提醒**：系统通过自动邮件或短信提醒读者归还逾期图书，管理员可以查看逾期情况并进行催还。

* **借阅历史与反馈**：

  * 记录每个读者的借阅历史，提供给读者查看自己的借阅记录。

  * 允许读者对已借阅的书籍进行评分和评价，帮助其他读者做出选择。

* **逾期滞纳金管理**：

  * 根据借阅规则计算逾期滞纳金，读者可以在线支付滞纳金，系统将自动更新滞纳金记录。

---

### 5. **报表与统计**：

* **借阅情况统计**：

  * 系统可以生成不同时间范围内的借阅统计报表（如日、周、月）。管理员可以查看哪些图书借阅量最高，哪些图书滞销。

  * **图书热度分析**：统计每本图书的借阅次数和评分，展示图书的受欢迎程度。

* **用户活动分析**：

  * 统计用户的借阅行为，包括借阅量、借阅频次、逾期情况等。

  * 根据用户行为生成活跃度报告，帮助管理员优化借阅规则和图书推荐。

* **滞纳金统计与管理**：

  * 自动生成滞纳金报表，显示各个读者的滞纳金支付情况，并允许管理员查看滞纳金的分布情况。

* **图形化报表展示**：

  * 使用图表（如柱状图、饼图、折线图等）展示借阅统计数据，帮助管理员直观地分析图书馆的运营情况。

---

### 6. **系统性能优化与安全控制**：

* **数据库优化**：

  * 为高频查询如借阅记录、图书信息和用户查询设计索引，提高查询效率。

  * 使用数据分区技术和缓存机制（如 Redis）来提高数据访问速度，减少数据库的负担。

* **数据备份与恢复**：

  * 定期备份图书馆的关键数据（如图书信息、借阅记录、用户数据），确保数据的安全性。

  * 支持快速数据恢复功能，以防止系统故障时数据丢失。

* **数据加密与安全**：

  * 对用户的敏感信息（如联系方式、支付记录等）进行加密存储。

  * 使用 SSL/TLS 加密数据传输，防止数据在传输过程中的泄露。

---

### 7. **图书馆座位预定与释放功能**：

为了提升图书馆的使用效率和管理，系统将增加 **座位预定** 和 **释放** 的功能，确保用户能够高效地利用可用位置。以下是功能设计和实现策略：

* **座位管理**：

  * **座位信息管理**：每个座位都有唯一标识符，并记录其位置、可用状态等信息。座位的状态可以是“空闲”或“已占用”。

  * **座位查询**：用户可以查询当前图书馆内可用的座位。系统根据座位状态和位置返回空闲座位的列表。

  * **座位预定**：读者可以预定空闲座位，预定成功后，座位的状态将变更为“已占用”，并将该座位的信息与用户的ID相关联。

  * **座位释放**：当读者离开图书馆时，他们需要手动释放座位。系统会更新该座位的状态为“空闲”，并移除读者与座位的绑定。

* **座位冲突管理**：

  * **防止重复预定**：系统会检查每个座位的状态，确保一个座位不会被多个读者预定。若用户尝试预定已占用座位，系统会返回错误信息。

---

### 8. **总结与优化方案**：

本系统不仅能实现图书馆的基本图书管理、借阅管理、用户权限管理等功能，还通过 **座位预定与释放功能**，提高了图书馆空间的利用率。此外，通过 **增量同步**、**实时同步** 和 **冲突解决机制** 等策略保证了系统的高可用性和数据一致性。

## 数据库 Tabel 设计：

### 1. **图书信息表 (Books)**

合并了图书分类信息，记录图书的基本信息。

| 字段名            | 数据类型         | 说明         |
| -------------- | ------------ | ---------- |
| book_id        | INT          | 主键，图书唯一标识符 |
| title          | VARCHAR(255) | 书名         |
| author         | VARCHAR(255) | 作者         |
| publisher      | VARCHAR(255) | 出版社        |
| isbn           | VARCHAR(20)  | ISBN号      |
| category       | VARCHAR(100) | 图书类别       |
| stock_quantity | INT          | 库存数量       |
| publish_date   | DATE         | 出版日期       |

* **规范化说明**：此表符合1NF，每个字段存储原子数据，且没有重复的列；2NF，所有非主键字段完全依赖于主键；3NF，字段之间没有传递依赖。

---

### 2. **用户信息表 (Users)**

合并了用户角色和权限信息，包含学生和管理员角色。

| 字段名               | 数据类型         | 说明           |
| ----------------- | ------------ | ------------ |
| user_id           | INT          | 主键，用户唯一标识符   |
| username          | VARCHAR(50)  | 用户名          |
| password_hash     | VARCHAR(255) | 密码哈希值        |
| email             | VARCHAR(255) | 邮箱           |
| role              | VARCHAR(20)  | 用户角色（学生/管理员） |
| registration_date | DATE         | 注册日期         |

* **规范化说明**：该表符合1NF，存储用户基本信息，2NF，角色与权限信息合并，3NF，没有冗余信息。

---

### 3. **借阅记录表 (Borrowing_Records)**

记录每个用户的借阅行为。

| 字段名          | 数据类型           | 说明              |
| ------------ | -------------- | --------------- |
| borrowing_id | INT            | 主键，借阅记录唯一标识符    |
| user_id      | INT            | 外键，关联 `Users` 表 |
| book_id      | INT            | 外键，关联 `Books` 表 |
| borrow_date  | DATE           | 借阅日期            |
| due_date     | DATE           | 应还日期            |
| return_date  | DATE           | 实际归还日期          |
| fine         | DECIMAL(10, 2) | 逾期滞纳金           |

* **规范化说明**：该表符合1NF，每个字段存储原子数据，2NF，借阅记录完全依赖于借阅记录ID；3NF，逾期滞纳金与借阅行为无传递依赖。

---

### 4. **同步日志表 (Sync_Logs)**

记录数据库同步的日志，支持跨库同步。

| 字段名           | 数据类型        | 说明                    |
| ------------- | ----------- | --------------------- |
| sync_id       | INT         | 主键，日志唯一标识符            |
| source_db     | VARCHAR(50) | 数据源数据库（MySQL、Oracle等） |
| target_db     | VARCHAR(50) | 目标数据库（MySQL、Oracle等）  |
| sync_status   | VARCHAR(20) | 同步状态（成功/失败）           |
| sync_time     | DATETIME    | 同步时间                  |
| error_details | TEXT        | 错误详情                  |

* **规范化说明**：符合1NF，日志条目没有重复数据；2NF，所有非主键数据依赖于主键；3NF，日志表与其他表之间没有冗余或传递依赖。

---

### 5. **逾期滞纳金表 (Overdue_Fines)**

记录每个读者的滞纳金状态。

| 字段名       | 数据类型           | 说明              |
| --------- | -------------- | --------------- |
| fine_id   | INT            | 主键，滞纳金唯一标识符     |
| user_id   | INT            | 外键，关联 `Users` 表 |
| amount    | DECIMAL(10, 2) | 滞纳金金额           |
| fine_date | DATE           | 滞纳金计算日期         |

* **规范化说明**：该表符合1NF，滞纳金信息与用户唯一标识相关，避免了重复数据；2NF，金额和计算日期依赖于滞纳金ID；3NF，没有传递依赖。

---

### 6. **座位信息表 (Seats)**

记录图书馆座位的状态和位置。

| 字段名           | 数据类型         | 说明           |
| ------------- | ------------ | ------------ |
| seat_id       | INT          | 主键，座位唯一标识符   |
| seat_location | VARCHAR(100) | 座位位置（如A区，B区） |
| status        | VARCHAR(20)  | 座位状态（空闲/已占用） |

* **规范化说明**：符合1NF，座位的唯一标识符和状态保持原子性；2NF和3NF，座位信息独立存储，没有冗余信息，避免了用户数据重复。

---

### 7. **座位预约表 (Seat_Reservations)**

记录读者的座位预约情况。

| 字段名              | 数据类型     | 说明               |
| ---------------- | -------- | ---------------- |
| reservation_id   | INT      | 主键，预约记录唯一标识符     |
| user_id          | INT      | 外键，关联 `Users` 表  |
| seat_id          | INT      | 外键，关联 `Seats` 表  |
| reservation_time | DATETIME | 座位预定时间           |
| release_time     | DATETIME | 释放时间（当用户离开时释放座位） |

* **规范化说明**：符合1NF，避免了重复的座位和预约信息；2NF，预约记录完全依赖于预约记录ID；3NF，表中没有传递依赖。

---

### 数据库设计规范化总结

1. **1NF**：所有表格字段只包含原子数据，没有重复数据列。
2. **2NF**：所有非主键字段完全依赖于主键，表中没有部分依赖。
3. **3NF**：没有传递依赖，每个非主键字段依赖于主键，不依赖其他非主键字段。

通过上述规范化过程，每个表的设计确保了数据的完整性、可维护性与可扩展性，避免了冗余数据的存储，并且符合业务需求的同时保证了数据一致性和高效查询。

## 在线图书馆管理系统技术报告

---

### **1. 实践目的**

本课程旨在使学生将数据库系统的基础理论知识应用于实际问题的分析与系统设计中。通过本次实践，学生需要独立完成一个完整的数据库应用系统，涵盖数据库设计、前端开发与应用交互、数据同步与管理等方面。实践的最终目标是让学生能够理解并实现数据库系统的设计、开发与优化。

---

### **2. 系统需求分析与总体设计**

#### **需求分析**

本在线图书馆管理系统旨在实现图书馆的基本功能，如图书管理、借阅管理、用户权限管理以及座位预定等。系统设计要求支持多数据库平台（MySQL、Oracle、SQL Server等），并实现跨库数据同步功能。主要功能模块包括：

* **图书信息管理**：新增、修改、删除图书信息，查询图书。
* **借阅管理**：借阅记录、逾期管理、滞纳金计算。
* **用户管理与权限控制**：读者、管理员角色的权限控制与管理。
* **座位预定与释放功能**：提供座位信息管理与座位预约功能。
* **报表与统计功能**：生成借阅统计、图书热度、滞纳金等数据报表。

#### **系统架构设计**

本系统采用了**客户端-服务器**架构，前端用户界面通过浏览器与后端的数据库管理系统进行交互。系统使用三种不同的数据库（如 MySQL、Oracle、SQL Server）来实现数据同步，确保不同数据库平台间的数据一致性。数据同步可通过实时同步与定期同步两种方式进行。

---

### **3. 数据库与同步机制设计**

#### **数据库设计**

数据库系统设计依据需求分析，涵盖了图书信息、用户信息、借阅记录、同步日志、滞纳金、座位信息及座位预约等表。所有表都严格遵循1NF、2NF、3NF的规范化要求，确保数据的完整性、可维护性和一致性。

以下是主要数据库表结构：

* **图书信息表（Books）**：包含图书的基本信息，如书名、作者、ISBN、分类、库存数量等。
* **用户信息表（Users）**：记录用户信息，包含用户名、密码哈希、角色信息（学生/管理员）等。
* **借阅记录表（Borrowing_Records）**：记录每个用户的借阅历史、借阅日期、应还日期及逾期滞纳金。
* **同步日志表（Sync_Logs）**：记录跨数据库同步操作的日志信息，确保同步的可靠性与可追溯性。
* **逾期滞纳金表（Overdue_Fines）**：记录每个读者的滞纳金金额和计算日期。
* **座位信息表（Seats）**：记录图书馆的座位状态，如空闲或已占用。
* **座位预约表（Seat_Reservations）**：记录读者的座位预约信息。

#### **数据同步机制**

系统需要支持多数据库同步，保证不同数据库中的数据保持一致。数据同步机制包括以下两种方式：

* **实时同步**：每当借阅记录、图书信息或用户数据发生变化时，立刻通过触发器（如 MySQL 的 `AFTER INSERT/UPDATE`）同步数据。
* **定期同步**：定时任务（如 cron 作业）定期执行数据同步，以减轻实时同步的压力。

#### **冲突检测与自动修复**

* **冲突检测**：当两个数据库中的数据发生冲突时（例如借阅记录不一致），系统会通过邮件自动通知管理员。
* **自动修复**：管理员可以选择手动修复冲突，或者系统根据预设规则自动选择更新最新的数据。

#### **同步性能优化**

* **增量同步**：每次同步仅传输变更数据，减少了数据传输量和带宽消耗。
* **数据去重**：确保每条记录在每个数据库中都是唯一的，避免重复数据导致同步失败。
* **同步日志**：记录每次同步操作的详细日志，提供审计功能。

---

### **4. 前端交互与权限控制设计**

#### **前端设计**

前端设计考虑到美观与易用性，界面设计清晰简洁，符合用户使用习惯。系统支持响应式设计，能够在不同分辨率下自适应显示，兼容主流浏览器（如 Chrome、Firefox、Edge）。功能模块通过清晰的导航栏进行区分，便于用户快速定位所需功能。

#### **系统交互与数据操作**

前端与后端通过 RESTful API 进行交互，系统支持对数据库的增、删、改、查等基本操作。所有操作结果都及时反馈给用户，并显示相关的提示信息，如操作成功、失败原因等。

#### **权限控制**

系统设计了细粒度的权限控制，区分了管理员和学生两种角色：

* **管理员**：具有最高权限，能够管理用户、图书、借阅记录、同步任务等。
* **学生**：只能进行查询借阅历史、查询图书信息、预约座位等操作。

角色权限管理采用了基于角色的访问控制（RBAC）模型。

---

### **5. 报表与统计功能设计**

#### **借阅情况统计**

系统可以生成借阅情况的报表，包括：

* 日、周、月等时间范围的借阅统计。
* 最受欢迎的图书和滞销图书。
* 图书的借阅量、评分等。

#### **滞纳金统计与管理**

系统自动生成滞纳金的报表，帮助管理员查看各个读者的滞纳金支付情况。

#### **图形化报表展示**

使用柱状图、饼图、折线图等图表方式展示数据，直观地帮助管理员分析系统运营情况。

### **6. 系统性能优化与安全控制**

#### **数据库优化**

* **索引设计**：为高频查询设计合适的索引，提升查询效率。
* **数据分区**：对大数据表进行分区，优化查询性能。
* **缓存机制**：使用 Redis 缓存机制，减少数据库的负担，提高访问速度。

#### **数据加密与安全**

* 对敏感用户数据进行加密存储，使用 SSL/TLS 加密传输数据。
* 实施数据库用户权限控制与访问审计，确保系统安全。

#### **数据备份与恢复**

* 定期备份数据库关键数据，并支持快速恢复，确保系统故障时数据不丢失。

### **7. 技术实现与优化建议**

#### **技术栈**

* **前端**：使用 HTML5、CSS3 和 JavaScript（React 或 Vue.js）进行开发。

* **负载均衡**：对于高并发的查询操作，采用负载均衡策略，确保系统的稳定性与高可用性。
* **移动端支持**：开发移动端应用，提升用户的操作体验，增加更多的便捷功能。
